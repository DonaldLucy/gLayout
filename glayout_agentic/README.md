# gLayout Agentic Workflow

This directory is a self-contained MVP for the workflow:

- input: natural language or an existing Python generator
- output: a runnable gLayout Python file
- validation: compile, optional execute, optional GDS existence check
- repair: feed failure logs back into the agent
- learning: store repair traces as JSONL so they can be fine-tuned later

It is intentionally built as a layer on top of the existing repo, not as a new model stack from scratch.

## Why this design

The repo already contains two important assets:

1. layout primitives and fixed topology generators such as `two_transistor_interdigitized`
2. an existing `llm-finetuning/` directory for compile-and-repair style experiments

So the fastest path is:

1. keep placement/routing encoded in Python generators
2. let the model write or repair those Python files
3. validate by running the generated file
4. turn failures into new supervised traces

That matches the two-layer parameter requirement:

- sizing parameters remain runtime arguments such as `width`, `length`, and `fingers`
- placement/routing choices remain fixed in the generated Python source

## Directory Layout

- `run_agent.py`: CLI entry point
- `check_env.py`: environment and dependency checker
- `workflow/`: agent loop, backends, prompts, validator, dataset logging
- `skills/`: deterministic skills/templates for tasks we want to guarantee
- `examples/`: known-good starter generators
- `training/train_qwen_lora.py`: single-GPU LoRA / QLoRA training script
- `data/training/`: JSONL traces generated by successful repair loops

## Hello World

The hello-world path is now intentionally split into three stages:

1. `glayout_agentic/examples/two_fet_separate.py`
2. `glayout_agentic/examples/two_fet_interdigitized.py`
3. `glayout_agentic/examples/two_fet_shared_diffusion.py`

The purpose of that split is to avoid conflating two different ideas:

- `interdigitized` means the two devices are alternated in placement
- `explicit merged diffusion` means the two devices share one physical diffusion region between them

The current Hello World used by the deterministic skill is stage 3:

- `glayout_agentic/examples/two_fet_shared_diffusion.py`

This version is built from a single two-finger primitive so the `left/shared/right`
diffusion terminals are explicit in the layout and in the top-level netlist.

The parameter split is:

- runtime: `width`, `length`, `fingers`
- fixed in source: device topology, shared-diffusion construction, tie strategy, and routing style

## Fresh A100 Setup

The commands below mirror the container flow you shared and add the agent-specific steps.

### 1. Start the Chipathon container

From the server:

```bash
cd ~/eda/designs/gLayout/docs/IIC-OSIC-TOOLS

  -e PYTHONPATH= \
  -e PYTHONHOME= \
  iic-osic-tools_chipathon_jupyter_uid_$(id -u) \
```

### 2. Activate the gLayout environment inside the container

```bash
export CONDA_ROOT=/headless/conda-env/miniconda3
source "$CONDA_ROOT/etc/profile.d/conda.sh"
conda activate GLdev
export PATH="$CONDA_PREFIX/bin:$CONDA_ROOT/condabin:/foss/tools/bin:/foss/tools/sak:/foss/tools/klayout:/foss/tools/libman:/foss/tools/osic-multitool:/foss/tools/rftoolkit/bin:/foss/tools/yosys/bin:/usr/local/sbin:/usr/local/bin:/usr/sbin:/usr/bin:/sbin:/bin"

cd /foss/designs/gLayout
export PYTHONPATH="$PWD/src"
export PDK_ROOT="/headless/conda-env/miniconda3/share/pdk"
export PDKPATH="$PDK_ROOT/sky130A"
```

### 3. Use the repo in-place, then install only the extra agent dependencies

```bash
cd /foss/designs/gLayout
export PYTHONPATH="$PWD/src"
python -m pip install -r glayout_agentic/requirements.txt
```

Inside the Chipathon container, `gLayout` is normally used in-place through
`PYTHONPATH="$PWD/src"`. Do **not** blindly run `pip install -e .` in the base
container environment unless you explicitly want to modify that environment.

The explicit `numpy<2` pin is important. In the local inspection for this task, the combination below failed during layout generation:

- `gdsfactory 7.7.0`
- `numpy 2.3.5`

The failure comes from `np.float_` removal in NumPy 2.x, so for this repo today the safe pin is still `numpy<2`.

### 4. Run the environment check

```bash
python glayout_agentic/check_env.py --strict
```

This checks:

- Python packages
- `magic`, `netgen`, `ngspice`, `klayout`, and `docker`
- `PDK_ROOT` and `PDKPATH`
- the known `gdsfactory` vs `numpy` compatibility issue

## Run the Hello World

Guaranteed path, using the deterministic skill:

```bash
python glayout_agentic/run_agent.py \
  --task "Generate a two FET merged diffusion SKY130 layout. width=2 length=0.15 fingers=1." \
  --backend skill \
  --execute \
  --run-drc-lvs \
  --output-py glayout_agentic/output/two_fet_shared_diffusion.py \
  --output-gds glayout_agentic/output/two_fet_shared_diffusion.gds
```

Recommended default path for day-to-day use:

```bash
python glayout_agentic/run_agent.py \
  --task "Generate a two FET merged diffusion SKY130 layout. width=2 length=0.15 fingers=1." \
  --backend auto \
  --execute \
  --run-drc-lvs
```

`auto` means:

- if a deterministic skill matches, use it first
- otherwise use the local Hugging Face backend

So for the current hello world, `auto` will still choose the deterministic skill
first. If you want to force real model generation with Qwen, use:

```bash
python glayout_agentic/run_agent.py \
  --task "Generate a two FET merged diffusion SKY130 layout. width=2 length=0.15 fingers=1." \
  --backend local-hf \
  --disable-skills \
  --model Qwen/Qwen2.5-Coder-7B-Instruct \
  --execute \
  --run-drc-lvs
```

`--disable-skills` is the strict mode for experiments where the code must come
only from the LLM:

- no template emission
- no skill-based code injection
- no skill hint appended to the model prompt

That is the safest way to demo the hello world while still keeping the model path available for harder tasks.

## Repair Existing Code

If you already have a generated Python file and want the loop to repair it:

```bash
python glayout_agentic/run_agent.py \
  --input-code broken_layout.py \
  --task "Repair this generator so it writes a valid GDS for a two FET shared diffusion cell." \
  --backend auto \
  --execute
```

Each run stores:

- prompts
- candidate code for every attempt
- validator logs
- structured attempt summaries

under `glayout_agentic/runs/<timestamp>/`.

## Self-Improvement Loop

When a run eventually succeeds, the workflow appends two JSONL files:

- `glayout_agentic/data/training/repair_traces.jsonl`
- `glayout_agentic/data/training/legacy_train_records.jsonl`

The first file is the richer agent trace for LoRA / QLoRA.
The second file is shaped to be compatible with the older compile-fix fine-tuning flow already living in `llm-finetuning/`.

So the long-term loop is:

1. let the model or skill attempt a task
2. run compile, execution, and optional DRC/LVS checks
3. capture failure logs
4. repair until success
5. append the successful trace to training data
6. periodically fine-tune a small coding model on the accumulated traces

## Model Recommendation For One 40GB A100

I do **not** recommend training from scratch. The best practical path here is PEFT / LoRA or QLoRA on a strong existing code model.

Recommended sequence:

1. `Qwen/Qwen2.5-Coder-7B-Instruct`
2. `Qwen/Qwen2.5-Coder-14B-Instruct` after the data pipeline is clean

Why:

- Official Qwen model cards say Qwen2.5-Coder is available in 0.5B, 1.5B, 3B, 7B, 14B, and 32B sizes, with 128K context support.
- The official 7B card lists about 7.61B parameters.
- The official 14B card lists about 14.7B parameters.
- Hugging Face PEFT and bitsandbytes docs explicitly support LoRA / QLoRA adapter training for this kind of workflow.

My recommendation for your exact hardware:

- Day 1: use `7B-Instruct + 4-bit QLoRA`
- Day 2+: try `14B-Instruct + 4-bit QLoRA` only after the pipeline is stable
- Avoid 32B on a single 40GB card for iterative self-repair training

This is partly an inference from the official model sizes and PEFT docs, not a hard vendor guarantee. The main tradeoff is iteration speed: for an agentic loop, a faster 7B model that you can run and retrain often usually beats a larger model that is slow to iterate.

References:

- [Qwen2.5-Coder-7B](https://huggingface.co/Qwen/Qwen2.5-Coder-7B)
- [Qwen2.5-Coder-7B-Instruct](https://huggingface.co/Qwen/Qwen2.5-Coder-7B-Instruct)
- [Qwen2.5-Coder-14B](https://huggingface.co/Qwen/Qwen2.5-Coder-14B)
- [Qwen2.5-Coder-14B-Instruct](https://huggingface.co/Qwen/Qwen2.5-Coder-14B-Instruct)
- [Transformers PEFT docs](https://huggingface.co/docs/transformers/main/en/peft)
- [Transformers bitsandbytes docs](https://huggingface.co/docs/transformers/v4.50.0/quantization/bitsandbytes)

## Fine-Tuning On The Collected Traces

After enough successful runs:

```bash
python glayout_agentic/training/train_qwen_lora.py \
  --dataset glayout_agentic/data/training/repair_traces.jsonl \
  --model Qwen/Qwen2.5-Coder-7B-Instruct \
  --output-dir glayout_agentic/models/qwen25_coder_7b_glayout_lora
```

This training script is intentionally scoped for a single A100 and uses:

- LoRA adapters
- optional 4-bit loading
- repair traces as supervised input/output pairs

## What This MVP Solves Right Now

- A new isolated directory for the agentic workflow
- A runnable hello-world target for explicit merged diffusion generation
- A progressive test path: separate -> interdigitized -> explicit merged diffusion
- A deterministic skill path so the first demo is reliable
- A local Hugging Face path for later model-driven generation
- Compile / execute / GDS validation
- Optional DRC / LVS validation in the agent loop with `--run-drc-lvs`
- Failure logging and JSONL dataset creation
- A Qwen-oriented LoRA training entry point
